🔌 API Endpoints

Contrato Backend (FastAPI) para el frontend del sistema de detección de anomalías

Base URL: http://localhost:8000/api/v1

📋 Para el desarrollador frontend

El backend expone endpoints REST para estado y consultas históricas, y un WebSocket para recibir alertas en tiempo real. Todos los datos de alertas siguen el esquema definido en alert_schema_contract.json (v1.1.0).
Los tipos de alerta son: vertical, alta_densidad, zona_muerta, cep_aterrizaje_emergencia.
Las severidades son: baja, media, alta, critica.
Los estados del ciclo de vida son: inicio, actualizacion, fin.

🚨 Alertas (REST)
GET /alerts Listar alertas con filtros y paginación

Devuelve las alertas activas, ordenadas por timestamp descendente. Soporta filtrado por tipo, severidad y paginación. Este es el endpoint principal para poblar el sidebar.

Query Parameters
ParámetroTipoDescripción
tipo opcional string Filtrar por tipo: vertical | alta_densidad | zona_muerta | cep_aterrizaje_emergencia
severidad opcional string Filtrar por severidad: baja | media | alta | critica
estado opcional string Filtrar por estado: inicio | actualizacion | fin
limit opcional int Máx. alertas a devolver. Default: 50, Máx: 200
offset opcional int Offset para paginación. Default: 0

⚡ WebSocket (Tiempo Real)
WS ws://localhost:8000/ws/alerts Stream de alertas en tiempo real

Conexión WebSocket que emite cada alerta nueva en tiempo real conforme el backend las consume del topic flight-alerts de Kafka. Este es el canal principal para actualizar el mapa y el sidebar sin polling.

Conexión
// JavaScript — conectar al WebSocket
const ws = new WebSocket("ws://localhost:8000/ws/alerts");

ws.onopen = () => {
  console.log("✅ WebSocket conectado");
};

ws.onmessage = (event) => {
  const alerta = JSON.parse(event.data);
  // alerta tiene el mismo schema que GET /alerts/{id}
  console.log("🚨 Nueva alerta:", alerta.tipo_alerta, alerta.severidad);
  
  // → Añadir tarjeta al sidebar
  // → Añadir marker al mapa
  // → Actualizar contadores
};

ws.onclose = () => {
  console.log("🔌 WebSocket desconectado — reconectando...");
  // Implementar reconnect con backoff exponencial
};

Reconexión recomendada

Si el WebSocket se cierra, implementar reconnect con backoff exponencial (1s, 2s, 4s, 8s... hasta 30s máx). Mientras esté desconectado, usar GET /alerts como fallback.

📐 Schema de alertas — Campos por tipo

⚠️ Regla clave

Todos los campos comunes están SIEMPRE presentes en cada alerta. Los campos opcionales pueden ser null — el frontend debe manejar nulls en la renderización. El campo tipo_alerta determina qué campos opcionales tendrán valor.
