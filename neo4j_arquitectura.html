1️⃣ ¿Por qué Neo4j?

Nuestro sistema genera alertas en tiempo real mediante Apache Flink, pero no las persiste ni las relaciona entre sí. Una base de datos relacional (SQL) podría almacenarlas, pero las consultas de relación cruzada — como "¿Qué otros aviones estaban en la misma celda H3 cuando se disparó esta alerta?" — requieren JOINs costosos que crecen exponencialmente con el volumen de datos.

Neo4j es una base de datos orientada a grafos que modela estas relaciones de forma nativa. En vez de tablas y JOINs, los datos se almacenan como nodos (entidades) conectados por relaciones (aristas), lo que permite recorrer millones de conexiones en milisegundos.

❌ SQL Tradicional

Requiere múltiples JOINs para responder: "Dame todos los vuelos que pasaron por la misma zona que el vuelo MOCK99 durante su alerta". Rendimiento O(n²) con tablas grandes.

✅ Neo4j (Grafos)

La misma consulta se resuelve recorriendo 2 relaciones desde el nodo Alerta. Rendimiento O(k) donde k = número de vecinos directos, independiente del tamaño total.

2️⃣ Modelo de Datos (Nodos y Relaciones)

Nodos

  • Nodo :Vuelo — Avión único identificado por icao24. Propiedades: callsign, origin_country
  • Nodo :Alerta — Anomalía detectada por Flink. Propiedades: tipo, severidad, timestamp, altitud
  • Nodo :ZonaH3 — Celda hexagonal H3 geográfica. Propiedades: celda, lat, lon

Relaciones

  • (:Vuelo)-[:TIENE_ALERTA]→(:Alerta) — Un vuelo genera una o más alertas a lo largo de su trayectoria
  • (:Vuelo)-[:VUELA_SOBRE]→(:ZonaH3) — Un vuelo atraviesa una o más zonas geográficas H3
  • (:Alerta)-[:OCURRE_EN]→(:ZonaH3) — Cada alerta se localiza espacialmente en una celda concreta

Diagrama del Grafo

    ┌──────────────┐          ┌──────────────┐          ┌──────────────┐
    │  ✈️  :Vuelo   │──TIENE──▶│  🚨 :Alerta  │──OCURRE─▶│  📍 :ZonaH3  │
    │  icao24      │  ALERTA  │  tipo        │    EN    │  celda       │
    │  callsign    │          │  severidad   │          │  lat / lon   │
    │  country     │          │  timestamp   │          │              │
    └──────┬───────┘          └──────────────┘          └──────────────┘
           │                                                   ▲
           └─────────── VUELA_SOBRE ───────────────────────────┘

3️⃣ Punto de Integración en la Arquitectura

Neo4j se integra en la Capa 4 (Backend / FastAPI). El mismo consumidor de Kafka que alimenta el WebSocket para el mapa en tiempo real, escribe simultáneamente en Neo4j usando el driver oficial de Python (neo4j). La escritura se realiza de forma asíncrona para no bloquear el flujo de alertas en tiempo real.

Además, el Backend expone endpoints REST adicionales que consultan Neo4j mediante el lenguaje Cypher para alimentar un panel de historial en el Frontend.

Flujo de Datos con Neo4j

  Flink → Kafka (flight-alerts) → FastAPI Consumer ──┬──▶ WebSocket → Mapa (tiempo real)
                                                      │
                                                      └──▶ Neo4j Driver → Persistir grafo
                                                                ↕
                                             FastAPI REST ← Cypher Queries ← Frontend Panel

4️⃣ Forma de Implementación

4.1 — Despliegue con Docker Compose

Se añade Neo4j como un servicio más en el docker-compose.yml existente:

  neo4j:
    image: neo4j:5-community
    ports:
      - "7474:7474"    # Neo4j Browser (interfaz web)
      - "7687:7687"    # Protocolo Bolt (driver Python)
    environment:
      NEO4J_AUTH: neo4j/opensky_secret
      NEO4J_PLUGINS: '["apoc"]'
    volumes:
      - neo4j_data:/data
    networks:
      - flink-net

4.2 — Código Python: Persistir Alertas en el Grafo

Dentro del Backend FastAPI, el consumidor de Kafka invoca esta función cada vez que recibe una alerta del topic flight-alerts: 

# pip install neo4j
from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://neo4j:7687", auth=("neo4j", "opensky_secret"))

def persistir_alerta(alerta: dict):
    """Crea o actualiza nodos Vuelo, Alerta y ZonaH3 con sus relaciones."""
    with driver.session() as session:
        session.run("""
            MERGE (v:Vuelo {icao24: $icao24})
            SET v.callsign = $callsign, v.origin_country = $pais

            CREATE (a:Alerta {
                tipo:      $tipo,
                severidad: $severidad,
                timestamp: datetime($ts),
                altitud:   $alt
            })

            MERGE (z:ZonaH3 {celda: $h3})
            SET z.lat = $lat, z.lon = $lon

            CREATE (v)-[:TIENE_ALERTA]->(a)
            CREATE (v)-[:VUELA_SOBRE]->(z)
            CREATE (a)-[:OCURRE_EN]->(z)
        """,
            icao24    = alerta["icao24"],
            callsign  = alerta["callsign"],
            pais      = alerta["origen_pais"],
            tipo      = alerta["tipo_alerta"],
            severidad = alerta["severidad"],
            ts        = alerta["timestamp"],
            alt       = alerta["altitud_actual"],
            h3        = alerta.get("celda_h3", ""),
            lat       = alerta["latitud"],
            lon       = alerta["longitud"]
        )

MERGE se usa para :Vuelo y :ZonaH3 para evitar duplicados (si el avión ya existe, lo reutiliza). CREATE se usa para :Alerta porque cada alerta es un evento único e irrepetible.

4.3 — Consultas Cypher de Ejemplo

// 1. Top 5 vuelos con más alertas críticas en los últimos 7 días
MATCH (v:Vuelo)-[:TIENE_ALERTA]->(a:Alerta)
WHERE a.severidad = "critica"
  AND a.timestamp > datetime() - duration("P7D")
RETURN v.callsign, COUNT(a) AS total
ORDER BY total DESC LIMIT 5
// 2. Todos los vuelos que estaban en la misma zona durante una anomalía
MATCH (a:Alerta)-[:OCURRE_EN]->(z:ZonaH3)<-[:VUELA_SOBRE]-(v:Vuelo)
WHERE a.timestamp > datetime() - duration("PT1H")
RETURN z.celda, COLLECT(DISTINCT v.callsign) AS vuelos_afectados
// 3. Vuelos reincidentes (mismo avión, múltiples alertas)
MATCH (v:Vuelo)-[:TIENE_ALERTA]->(a:Alerta)
WITH v, COUNT(a) AS alertas WHERE alertas > 3
RETURN v.icao24, v.callsign, alertas
ORDER BY alertas DESC

5️⃣ Valor Añadido de Neo4j al Sistema

🔍 Análisis Forense

Reconstruir la secuencia completa de eventos alrededor de una anomalía navegando las relaciones del grafo en ambas direcciones.

🔁 Detección de Patrones

Identificar aerolíneas, rutas o aviones específicos con comportamiento anómalo recurrente a lo largo del tiempo.

🗺️ Correlación Espacial

Descubrir "zonas calientes" donde se concentran las alertas usando las celdas H3 como nodos geográficos del grafo.

☁️ Escalabilidad

Neo4j Aura (SaaS) permite escalar la capa de persistencia sin gestionar infraestructura propia. Compatible con Kubernetes.
